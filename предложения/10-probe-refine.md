# Предложение 10: Probe-and-Refine -- самосовершенствование агента

**Дата:** 2026-06-25
**Этап Крепости:** Evolution
**Приоритет:** Средний
**Статус:** Ожидает рассмотрения

---

## Проблема

Агент Крепости использует статические инструкции (системные промпты, guidance-файлы). Эти инструкции:

- Написаны человеком и отражают его понимание задачи, а не реальный опыт агента
- Не адаптируются к новым типам запросов и ошибок
- Не улучшаются на основе прошлых успехов/неудач
- Требуют ручного обновления при изменении кодовой базы

При этом Крепость уже содержит в дорожной карте этап Evolution (Фаза 2) -- контролируемое самосовершенствование агента. Probe-and-Refine предоставляет конкретный, проверенный механизм для этого.

## Решение: Probe-and-Refine Tuning

Процедура автоматического улучшения файла инструкций агента через синтетические bug-fix пробы. Агент сам диагностирует и патчит свои операционные инструкции, без дорогого agent-loop и tool-use во время тюнинга.

**Источник:** [arXiv:2606.20512](https://arxiv.org/abs/2606.20512v1) -- "Probe-and-Refine Tuning of Guidance for Coding Agents"

**Ключевой вывод исследования:** важно не НАЛИЧИЕ гайданса, а КАК он порождён. Статическая база знаний даёт 28.3% resolve rate; Probe-and-Refine даёт 33.0% (SWE-bench Verified, Qwen3.5-35B-A3B, 200 шагов, p<0.001). Прирост от coverage (+14.5pp evaluable patches), не от precision (~59% константа).

## Механизм работы

### Принцип

Модель генерирует синтетические bug-fix пробы, пропускает их через одношаговые LLM-вызовы, анализирует результаты и итеративно правит свой guidance-файл. Решающая переменная -- КАК произведён гайданс, а не сам факт его наличия.

### Шаги

1. **Генерация синтетических проб (Probe Generation)**
   - Из кодовой базы извлекаются реальные файлы
   - Генерируются синтетические баги (по паттернам реальных ошибок)
   - Каждая проба: (файл с багом, описание бага, эталонный фикс)

2. **Одношаговая диагностика (Single-Step LLM Calls)**
   - Агент получает пробу + текущий guidance-файл
   - Пытается диагностировать и исправить баг за один шаг
   - БЕЗ agent-loop, БЕЗ tool-use -- чистый LLM-вызов

3. **Анализ и патч гайданса (Refine)**
   - Сравнение результата агента с эталонным фиксом
   - Диагностика: что в гайдансе помогло / помешало
   - Итеративный патч guidance-файла

4. **Верификация**
   - Повторный прогон проб с обновлённым гайдансом
   - Оценка: улучшился ли resolve rate
   - Откат при деградации

### Диаграмма рабочего процесса

```
+================================================================+
|              PROBE-AND-REFINE: РАБОЧИЙ ПРОЦЕСС                  |
+================================================================+

                    +---------------------------+
                    |    Кодовая база Крепости   |
                    |    (Python, YAML, конфиги) |
                    +-------------+-------------+
                                  |
                                  v
                    +---------------------------+
                    |  1. ГЕНЕРАЦИЯ ПРОБ         |
                    |                           |
                    |  Извлечь реальные файлы    |
                    |  Внести синтетические баги |
                    |  Сформировать пары:        |
                    |  (баг, эталонный фикс)     |
                    +-------------+-------------+
                                  |
                                  | N проб
                                  v
              +-----------------------------------+
              |  2. ОДНОШАГОВАЯ ДИАГНОСТИКА        |
              |                                   |
              |  Для каждой пробы i=1..N:          |
              |                                   |
              |  +-----------------------------+  |
              |  | LLM-вызов (один шаг):       |  |
              |  |                             |  |
              |  | Вход:                        |  |
              |  |   - Проба (файл с багом)     |  |
              |  |   - Текущий guidance-файл    |  |
              |  |                             |  |
              |  | Выход:                       |  |
              |  |   - Диагноз + предложенный   |  |
              |  |     фикс                     |  |
              |  +-----------------------------+  |
              +-------------------+---------------+
                                  |
                                  | N результатов
                                  v
              +-----------------------------------+
              |  3. АНАЛИЗ РЕЗУЛЬТАТОВ             |
              |                                   |
              |  Сравнить с эталонными фиксами:    |
              |                                   |
              |  Проба 1: УСПЕХ -- гайданс помог   |
              |  Проба 2: ПРОВАЛ -- не нашёл файл  |
              |  Проба 3: ПРОВАЛ -- неверный паттерн|
              |  Проба 4: УСПЕХ                    |
              |  ...                               |
              +-------------------+---------------+
                                  |
                                  v
              +-----------------------------------+
              |  4. ПАТЧ ГАЙДАНСА (REFINE)         |
              |                                   |
              |  LLM анализирует паттерны          |
              |  неудач и генерирует патч:          |
              |                                   |
              |  "Добавить правило: при ошибке в   |
              |   pipeline.py сначала проверить    |
              |   импорты в normalize.py"           |
              |                                   |
              |  guidance_v1 --> guidance_v2        |
              +-------------------+---------------+
                                  |
                                  v
        +--------------------------------------------+
        |  5. ВЕРИФИКАЦИЯ (Human Approval Gate)       |
        |                                            |
        |  +-- Автоматическая проверка:              |
        |  |   Повторный прогон проб с guidance_v2   |
        |  |   Resolve rate: 28.3% --> 33.0%?        |
        |  |                                          |
        |  +-- Человеческое одобрение:               |
        |  |   Владелец читает diff гайданса         |
        |  |   Решение: ОДОБРИТЬ / ОТКЛОНИТЬ         |
        |  |                                          |
        |  +-- При деградации:                       |
        |      ОТКАТ к guidance_v1                   |
        +--------------------------------------------+
                    |                    |
                    v                    v
            +-------------+     +---------------+
            |  ОДОБРЕНО   |     |  ОТКЛОНЕНО    |
            |             |     |               |
            |  guidance_v2|     |  guidance_v1  |
            |  --> prod   |     |  остаётся     |
            +-------------+     +---------------+
                    |
                    | Следующая итерация
                    v
              (возврат к шагу 1)
```

## Контролируемое самосовершенствование (Human Approval Gate)

Anthropic (2026) предупреждает об опасностях неконтролируемого recursive self-improvement (RSI). В контексте Крепости это критично: агент, свободно переписывающий свои промпты, может деградировать или выйти из-под контроля.

**Источник:** [Anthropic RSI Warning 2026](https://medium.com/@deekshithamiddhla30818/the-most-important-ai-warning-of-2026-anthropics-concerns-about-self-improving-ai-fe0b32b52677)

### Многоуровневая защита

```
Уровень 1: Автоматическая проверка
  - Resolve rate не упал? Coverage не снизился?
  - Регрессионные тесты пройдены?
  - Нет запрещённых паттернов в новом гайдансе?

Уровень 2: Sovereign Execution Broker (предложение 07)
  - Мутация гайданса -- это execution, требующая сертификата
  - Scoped execution identity для операции патча
  - Аудит-лог каждого изменения с подписью

Уровень 3: Человеческое одобрение
  - Diff нового гайданса показывается владельцу
  - Владелец принимает решение: ОДОБРИТЬ / ОТКЛОНИТЬ / ДОРАБОТАТЬ
  - Без одобрения изменение НЕ применяется в production

Уровень 4: Откат
  - Все версии гайданса хранятся в git
  - Откат к любой предыдущей версии за одну команду
  - Автоматический откат при падении resolve rate ниже baseline
```

## Интеграция с дорожной картой Крепости (Evolution, Фаза 2)

### Связь с существующими технологиями

| Технология из дайджестов | Роль в Probe-and-Refine |
|---|---|
| ExpRL (arXiv:2606.17024) | Dense rewards на промежуточные шаги -- аналог оценки частичных фиксов |
| PACT: Plan, Align, Commit, Think (arXiv:2606.16995) | Паттерн верификации перед исполнением -- валидация патча гайданса |
| Multi-agent consensus (Cloudflare) | Несколько агентов-верификаторов проверяют патч гайданса |
| LLM-as-judge с калибруемой рубрикой | Оценка качества диагностики и фиксов по рубрике |
| Multi-LCB benchmark (arXiv:2606.20517) | Метрика для оценки регрессий при самосовершенствовании |
| Sovereign Execution Brokers (arXiv:2606.20520) | Certificate-bound контроль мутаций гайданса |

### Место в архитектуре Evolution

```
Evolution Фаза 1 (текущая):
  - Статические промпты/гайданс
  - Ручное обновление

Evolution Фаза 2 (данное предложение):
  - Probe-and-Refine для автоматического улучшения гайданса
  - Human approval gate
  - Аудит-лог всех изменений

Evolution Фаза 3 (будущее):
  - Полный self-improvement loop с multi-agent consensus
  - Автоматическая генерация проб из production-ошибок
  - Continuous improvement с минимальным участием человека
```

## Результаты из исследования (arXiv:2606.20512)

| Конфигурация | Resolve Rate | Evaluable Patches |
|---|---|---|
| Без гайданса | 25.5% | Baseline |
| Статическая база знаний | 28.3% | +8.2pp |
| Probe-and-Refine (200 шагов) | **33.0%** | **+14.5pp** |
| p-value | < 0.001 | |

**Эквивалентный прирост:** 27x speedup на SWE-bench Verified (по метрике resolve rate / compute cost, так как одношаговые LLM-вызовы без agent-loop многократно дешевле полного агентного цикла).

**Важное ограничение:** деградирует, если модель не генерирует достаточно диагностичный вывод. Требуется модель с хорошим reasoning (Qwen3.5-35B+, или аналогичная).

## Этапы интеграции

### Фаза 1: Прототип (2 недели)
- Генератор синтетических проб из кодовой базы Крепости
- Одношаговая диагностика на текущей модели
- Простой патч-механизм для guidance-файла
- Метрики: resolve rate, coverage

### Фаза 2: Верификация (1 неделя)
- Human approval gate (diff в Telegram/CLI)
- Автоматический откат при деградации
- Регрессионные тесты
- Интеграция с git (версионирование гайданса)

### Фаза 3: Интеграция с SEB (1 неделя)
- Мутация гайданса через Sovereign Execution Broker (предложение 07)
- Audit-log с подписями
- Scoped execution identity

### Фаза 4: Production (2 недели)
- Запуск на реальной кодовой базе Крепости
- Мониторинг resolve rate в динамике
- Калибровка частоты итераций (ежедневно / еженедельно)
- Документация процесса для владельца

## Риски и митигация

| Риск | Митигация |
|---|---|
| Неконтролируемое RSI (recursive self-improvement) | Human approval gate + SEB + автоматический откат |
| Деградация гайданса | Регрессионные тесты + baseline resolve rate + откат |
| Агент "обманывает" метрики | Независимые тесты, не входящие в training set проб |
| Переоптимизация под синтетические баги | Периодическая валидация на реальных issues |
| Вычислительная стоимость | Одношаговые LLM-вызовы, не agent-loop (в десятки раз дешевле) |
