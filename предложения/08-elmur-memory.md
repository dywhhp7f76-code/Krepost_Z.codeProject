# Предложение 08: Архитектура долговременной памяти ELMUR

**Дата:** 2026-06-25
**Этап Крепости:** Memory
**Приоритет:** Высокий
**Статус:** ✅ ОДОБРЕНО (2026-07-16) — оператор одобрил; порт EpisodicMemory в `krepost/memory/episodic.py` выполнен, полная интеграция в orchestrator — следующий этап.

---

## Проблема

Текущая память агента Крепости ограничена контекстным окном трансформера. Внимание (attention) имеет квадратичную сложность O(n^2) по длине последовательности. Наивное увеличение контекста:

- Экспоненциально увеличивает стоимость инференса
- Не масштабируется для долгосрочных задач
- Теряет ранние воспоминания при переполнении окна
- Не позволяет накапливать опыт между сессиями

## Решение: ELMUR (External Long-term Memory for Universal Reasoning)

Архитектура ELMUR (ICLR 2026, AIRI) предлагает **внешний слой памяти, отделённый от контекстного окна**, что устраняет квадратичный рост и обеспечивает пожизненное хранение состояний.

**Источник:** [AIRI / ICLR 2026](https://habr.com/ru/companies/airi/articles/1047598/)

---

## Ключевые компоненты

### 1. Внешний слой памяти (External Memory Layer)

- Память хранится вне контекстного окна трансформера
- Обращение к памяти через механизм retrieval, а не через attention
- Сложность обращения к памяти: O(n) вместо O(n^2)
- Поддержка произвольного объёма хранимых эпизодов

### 2. Эпизодическая память (Episodic Memory)

- Каждое взаимодействие записывается как эпизод с метаданными
- Эпизоды: пользовательские запросы, ответы агента, результаты tool-вызовов, ошибки
- Индексация по семантическому содержанию и временным меткам
- Связь с заметками Obsidian как внешним хранилищем знаний

### 3. Механизм забывания с весами важности (Importance-Weighted Forgetting)

- Каждому воспоминанию присваивается вес важности (importance score)
- Веса затухают со временем (exponential decay)
- Часто запрашиваемые воспоминания укрепляются (reinforcement)
- Механизм аналогичен консолидации памяти в нейронауке
- Порог забывания настраивается администратором

### 4. Интеграция с Hopfield Memory (из Крепости)

Крепость уже использует концепцию Modern Hopfield Network (FlowEdit, arXiv:2606.20518) как content-addressable episodic memory. ELMUR дополняет этот подход:

- Hopfield-сеть обеспечивает fuzzy retrieval через soft attention с similarity gate
- ELMUR добавляет долгосрочное хранение поверх Hopfield-адресации
- Коррекции хранятся как latent conditioning edits (в пространстве embedding)
- Совместная работа: Hopfield для быстрого поиска, ELMUR для персистентного хранения

## Сравнение архитектур

### Текущая архитектура памяти Крепости

```
+--------------------------------------------------+
|               ТЕКУЩАЯ АРХИТЕКТУРА                 |
+--------------------------------------------------+
|                                                    |
|  Пользователь --> Запрос                           |
|       |                                            |
|       v                                            |
|  +------------------------------------------+     |
|  | Контекстное окно трансформера              |     |
|  | (вся память живёт здесь)                   |     |
|  |                                            |     |
|  |  [Системный промпт]                        |     |
|  |  [История диалога]     <-- O(n^2) attention |     |
|  |  [Retrieved chunks]                        |     |
|  |  [Tool results]                            |     |
|  |                                            |     |
|  +------------------------------------------+     |
|       |                                            |
|       v                                            |
|  SmartCache (L1/L2/L3)                             |
|  - L1: QueryEmbeddingCache (exact match)           |
|  - L2: RAGResultsCache (cosine >= 0.92)            |
|  - L3: LLMResponseCache (exact match)              |
|       |                                            |
|       v                                            |
|  ChromaDB / LanceDB (векторное хранилище)          |
|  - Без механизма забывания                         |
|  - Без весов важности                              |
|  - Без консолидации                                |
|                                                    |
+--------------------------------------------------+

Ограничения:
- Контекст растёт --> attention O(n^2) --> рост стоимости
- Нет персистентной эпизодической памяти между сессиями
- Нет приоритизации воспоминаний
- Нет механизма забывания устаревшей информации
```

### Архитектура с ELMUR

```
+--------------------------------------------------+
|           АРХИТЕКТУРА С ELMUR                      |
+--------------------------------------------------+
|                                                    |
|  Пользователь --> Запрос                           |
|       |                                            |
|       v                                            |
|  +------------------------------------------+     |
|  | Контекстное окно (компактное)              |     |
|  |                                            |     |
|  |  [Системный промпт]                        |     |
|  |  [Текущий диалог]     <-- O(k^2), k << n   |     |
|  |  [ELMUR-retrieved]                         |     |
|  |  [Tool results]                            |     |
|  +------------------------------------------+     |
|       |              ^                             |
|       |              | retrieval O(n)              |
|       v              |                             |
|  +------------------------------------------+     |
|  | ELMUR: Внешний слой памяти                 |     |
|  |                                            |     |
|  |  +-- Hopfield Network (fuzzy retrieval) --+|     |
|  |  |   soft attention + similarity gate      ||     |
|  |  +----------------------------------------+|     |
|  |                                            |     |
|  |  +-- Importance-Weighted Store -----------+|     |
|  |  |   [Эпизод 1] w=0.95 (свежий, частый)   ||     |
|  |  |   [Эпизод 2] w=0.72 (средний)          ||     |
|  |  |   [Эпизод 3] w=0.15 (старый, редкий)   ||     |
|  |  |   [Эпизод 4] w=0.03 --> ЗАБЫТ          ||     |
|  |  +----------------------------------------+|     |
|  |                                            |     |
|  |  +-- Latent Corrections ------------------+|     |
|  |  |   Коррекции как embedding-правки        ||     |
|  |  |   (без дообучения модели)               ||     |
|  |  +----------------------------------------+|     |
|  +------------------------------------------+     |
|       |                                            |
|       v                                            |
|  SmartCache (L1/L2/L3) -- без изменений            |
|       |                                            |
|       v                                            |
|  ChromaDB / LanceDB + Obsidian                     |
|  (хранилище фактов, не эпизодов)                   |
|                                                    |
+--------------------------------------------------+

Преимущества:
- Контекст фиксированного размера k --> attention O(k^2)
- Память растёт линейно, не квадратично
- Персистентная эпизодическая память между сессиями
- Автоматическое забывание неважного
- Укрепление часто используемых воспоминаний
```

## Связь с существующими компонентами Крепости

| Компонент Крепости | Роль в интеграции с ELMUR |
|---|---|
| SmartCache (L1/L2/L3) | Остаётся как есть. ELMUR работает на уровне выше -- управляет какие эпизоды попадают в контекст |
| ChromaDB/LanceDB | Хранилище фактов (knowledge base). ELMUR хранит эпизоды (опыт агента) |
| Security Pipeline (4 слоя) | Фильтрация ДО записи в ELMUR. Verdict=RED/YELLOW не записывается |
| KVEraser (arXiv:2606.17034) | Дополняет ELMUR: удаление вредоносных инъекций из KV-cache в runtime |
| OCC-RAG (AIRI, 0.6B/1.7B) | Идеальный reader поверх ELMUR-retrieved эпизодов -- строго по контексту |

## Этапы интеграции

### Фаза 1: Прототип (2-3 недели)
- Реализация хранилища эпизодов с весами важности
- Механизм затухания (exponential decay с настраиваемым периодом полураспада)
- API: `store_episode()`, `retrieve_relevant()`, `forget_below_threshold()`
- Тесты на синтетических данных

### Фаза 2: Hopfield-адресация (2 недели)
- Интеграция Modern Hopfield Network для content-addressable retrieval
- Similarity gate для fuzzy matching
- Бенчмарк: сравнение с чистым cosine-поиском по ChromaDB

### Фаза 3: Интеграция с пайплайном (2 недели)
- Подключение к Security Pipeline (запись только GREEN-вердиктов)
- Подключение к SmartCache (ELMUR как источник контекста для L2/L3)
- Подключение к Obsidian (синхронизация между заметками и эпизодической памятью)

### Фаза 4: Настройка и оптимизация (1 неделя)
- Калибровка порогов забывания
- Оптимизация размера retrieved-контекста
- A/B тестирование качества ответов с/без ELMUR

## Риски и митигация

| Риск | Митигация |
|---|---|
| Утечка PII через эпизодическую память | Layer 4 (PII masking) фильтрует ДО записи в ELMUR |
| Отравление памяти (adversarial episodes) | Security Pipeline (Layer 1-3) отсеивает вредоносный ввод |
| Чрезмерное забывание важной информации | Ручная маркировка "защищённых" эпизодов (pinned memories) |
| Рост хранилища | Периодическая компактификация + настраиваемые лимиты |
