# KVEraser Paper

Исследование по KVEraser и интеграции в систему КРЕПОСТЬ.

## Проблема: стоимость очистки KV-cache

При обнаружении harmful-контента в сгенерированном ответе стандартный подход — полный пересчёт KV-cache без вредоносных токенов. Однако full recompute даёт катастрофический прирост латентности: **+1760%** по сравнению с обычным инференсом. Для production-системы с требованиями к latency это неприемлемо.

## KVEraser: обучаемое удаление harmful spans

KVEraser предлагает элегантное решение: вместо пересчёта всего кеша производится **точечное стирание** (erasure) только тех токенов, которые содержат harmful-контент. Результат: прирост латентности всего **+24%** — в 73 раза меньше, чем при полном пересчёте.

## Механизм работы

1. **Идентификация harmful spans:** KVEraser использует **binary mask на attention heads** для определения, какие токены в KV-cache содержат вредоносную информацию. Маска обучается на размеченных данных с adversarial-примерами.

2. **Точечное стирание:** идентифицированные токены обнуляются в KV-cache без пересчёта остальных значений. Attention-механизм модели автоматически перераспределяет вес на оставшиеся (безопасные) токены.

3. **Сохранение когерентности:** благодаря тому, что стираются только harmful spans, а не весь контекст, модель сохраняет когерентность ответа и контекстуальную память о безопасной части диалога.

## Применение в Layer 4 (Post-Processor)

В архитектуре КРЕПОСТИ KVEraser интегрирован в **Layer 4 — Output Post-Processor**:

- После генерации ответа модели Post-Processor анализирует выходные токены
- При обнаружении harmful spans (через Guard-классификатор) KVEraser стирает соответствующие записи из KV-cache
- Модель перегенерирует только затронутую часть ответа, а не весь текст
- Это позволяет сохранить приемлемую латентность даже при срабатывании постфильтрации

## Сравнение подходов

| Подход | Латентность | Приемлемость |
|--------|------------|-------------|
| Без очистки | Базовая | Небезопасно |
| Full recompute | +1760% | Неприемлемо для production |
| KVEraser | +24% | Приемлемо |

## Альтернатива

Полный пересчёт KV-cache остаётся как fallback-механизм на случай, если KVEraser не может корректно идентифицировать harmful spans (например, при новом типе атаки). Однако в штатном режиме этот подход не используется из-за неприемлемой задержки.
