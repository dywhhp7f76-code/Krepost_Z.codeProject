# 🧪 Тренировочная песочница — архитектурная карта v1.1

Разбор ДО кода. Цель: Guard учится отличать атаки на размеченных тобой
примерах через **few-shot память** (вариант а — без дообучения весов).

9 камней (6 моих + 3 твоих) + hardening из 6 внешних аудитов (см. раздел v1.1 внизу).

---

## 🔒 ГЛАВНЫЙ ПРИНЦИП: физический air-gap

Грязное (яды, разметка, генерация) живёт на **съёмном SSD** (ACASIS/WD Blue).
Диск воткнут → работаешь с полигоном. Диск вынут → боевая система стерильна,
физически не может дотянуться до ядов.

```
  СЪЁМНЫЙ SSD (воткнут только когда тренируешь)        MAC STUDIO (боевой, всегда)
  ┌─────────────────────────────────────┐             ┌──────────────────────────┐
  │  training/                          │             │  Guard (Qwen3Guard-Gen-4B) │
  │   ├── _incoming/  (автоген, карантин)│   выжимка   │   + few-shot база        │
  │   ├── clean/      (чистые примеры)   │  ──эмбед──> │   (эмбеддинги атак,      │
  │   └── poison/     (заражённые)       │   под gate  │    НЕ тексты ядов)       │
  │                                     │             │                          │
  │  training_labels.db (разметка)      │             │  security.py (боевой)    │
  │  training_index/   (свой вектор-БД) │             │  VerdictLog (боевой лог) │
  └─────────────────────────────────────┘             └──────────────────────────┘
         ↑ всё грязное здесь                                  ↑ сюда едут только числа
```

**Что переходит границу:** НЕ тексты ядов, а только **эмбеддинги** (числа-координаты
известных атак) + пороги. Как фоторобот вместо живого преступника. Сам яд остаётся
на диске, уезжает с ним. Guard в бою сравнивает ввод с фотороботами.

## Модуль 1. Структура полигона (на съёмном SSD)

```
training/
  _incoming/     # автоген (Garak, attacker) валит СЮДА — карантин [Камень 6]
                 # в рабочий датасет переходит только после твоего просмотра
  clean/         # чистые примеры — эталон нормы
  poison/        # заражённые примеры — образцы атак
```

**Камень 6 (контроль автогена):** Garak/attacker не валят прямо в clean/poison.
Только в `_incoming/`. Ты просматриваешь → подтверждаешь → пример переезжает в
clean/ или poison/. Тот же gate, что везде: автомат предлагает, ты принимаешь.

---

## Модуль 2. Разметка (training_labels.db — отдельная таблица, на SSD)

Метка хранится **снаружи файла**, в БД по хешу [Камень 3]. НЕ во frontmatter —
иначе повторяем дыру frontmatter-injection (заражённый файл соврёт про метку).

Отдельно от VerdictLog [твоё решение]: боевой лог и учебная коллекция — разное.
VerdictLog = что Guard видел в бою. training_labels = что ты заготовил для учёбы.

```
training_labels:
  example_hash   TEXT PK    # sha256 тела примера
  label          TEXT       # 'poison' | 'clean'
  source         TEXT       # 'manual' | 'garak' | 'attacker'
  added_date     REAL       # [Камень 5] для отслеживания свежести/дрейфа
  note           TEXT       # твой коммент (опц.)
```

---

## Модуль 3. Изолированный векторный индекс (training_index/ на SSD)

[Камень 1 — критический] Заражённые примеры эмбеддятся в **отдельный** индекс,
НЕ в боевую память. Боевой recall сюда не заглядывает НИКОГДА. Физически другое
хранилище (как attack_vault). Иначе отрава протечёт в обычный поиск и всплывёт
в ответе основного мозга.

---

## Модуль 4. Двойная проверка чистых [Камень 4]

Контринтуитивный, но важный. Когда заносишь пример в `clean/`:
- прогнать его через Guard ПЕРЕД занесением;
- если Guard на «чистом» примере СРАБОТАЛ → флаг тебе: «ты уверен, что оно чистое?»

Ловит ТВОИ ошибки разметки. Яд, который ты не распознал и пометил как чистый —
это хуже явного яда: ты сам научишь Guard пропускать атаку как норму. Двойная
проверка именно на чистой стороне это ловит.

---

## Модуль 5. Сборка выжимки (под gate, пересекает границу)

Когда диск воткнут, под твоим подтверждением:
1. берём poison-примеры из индекса;
2. достаём их **эмбеддинги** (числа, не тексты);
3. кладём пороги (насколько близко = «похоже на атаку»);
4. пишем это в боевую few-shot базу на Mac Studio.

Тексты ядов НЕ переходят. Только координаты. Вынул диск — Guard работает по
выжимке, сами яды отрезаны.

---

## Модуль 6. Боевое применение (на Mac Studio, диск НЕ нужен)

Few-shot встраивается в существующий security.py как ещё один сигнал:
- приходит ввод → Guard слой 1 (regex) + слой 2 (Qwen3Guard-Gen-4B) + **few-shot match**;
- few-shot: близок ли ввод к известным атакам из выжимки? (косинус);
- близок → поднять подозрение (в YELLOW/RED).

Это НЕ замена Guard, а добавочный сигнал. Слой памяти поверх слоёв проверки.

---

## 🌊 ПОТОК ДАННЫХ (полный цикл)

```
1. Ты/Garak/attacker → training/_incoming/        (диск воткнут)
2. Ты просматриваешь → clean/ или poison/         (gate 1: что попадает в датасет)
3. Разметка → training_labels.db                  (по хешу, не в файле)
4. clean-пример → прогон через Guard              (Камень 4: ловит ошибки разметки)
5. poison → эмбеддинг → training_index/           (изолированный индекс)
6. Сборка выжимки → эмбеддинги на Mac Studio       (gate 2: что пересекает границу)
7. Вынул диск.
8. Боевой Guard сравнивает ввод с выжимкой         (few-shot, диск не нужен)
```

Два gate: (1) что попадает в датасет, (2) что пересекает воздушный зазор.

## 🧬 Защита от collapse (урок Anima)

Few-shot НЕ дообучает веса → катастрофического забывания нет by design.
Но дрейф датасета есть [Камень 5]: старые атаки устаревают, копятся дубли.
`added_date` позволит потом пересматривать свежесть. Внешняя новизна (Garak,
твои новые яды) — это и есть приток энтропии, который не даёт системе
схлопнуться в свой стиль. Полигон сам по себе — источник новизны для Guard.

## Модуль 7. Валидация при импорте [твоя находка]

Проверка тройки в момент занесения примера — иначе база накопит шум:

1. **Хеш** — sha256 тела ПОСЛЕ нормализации (NFKC, как в security). Иначе
   тот же яд с лишним пробелом = другой хеш = дубль-призрак.
2. **Метка** — строго из набора {poison, clean}. Опечатка → отказ (как Pydantic).
3. **Конфликт** — если хеш уже есть с ДРУГОЙ меткой → **СТОП, спросить тебя**
   [твоё решение]. Новая метка побеждает только после подтверждения. Молча не
   перезаписываем — смена метки это сигнал (ошибся раньше / сейчас / коллизия).

Gate на человеке на уровне импорта.

---

## Модуль 8. Версионирование разметки [твоя находка]

`training_labels.db` — твой накопленный труд (месяцы разметки). Жалко потерять.
Попадает под механизм «старой кожи»:

- перед коммитом — **дамп в JSONL** (построчно, git видит каждую смену метки →
  читаемый журнал твоих решений о ядах) [твоё решение];
- **git-crypt** шифрует — это карта твоих атак, в открытом виде чувствительна
  (чужой прочитает, что защита знает, а что нет);
- откат к разметке любой давности, если напортачил.

Прозрачно для тебя, закрыто для чужого.

---

## Модуль 9. Метрика качества few-shot [твоя находка — критическая]

Без неё песочница СЛЕПАЯ — тренируешь, не зная, стало ли лучше.

**Held-out poison recall:**
- яды делим: 70% обучающая (в выжимку) / 30% отложенная (Guard НЕ видел) [твоё];
- проверяем Guard на отложенной: recall = пойманные / всего отложенных;
- held-out имитирует НОВУЮ атаку → высокий recall = few-shot обобщает, а не зубрит.

**Split стратифицированный по типу атаки** [дополнение]: каждый тип (инъекции,
джейлбрейки, encoding-обход) делим 70/30 отдельно. Иначе все инъекции в обучение,
все джейлбрейки в held-out → recall покажет «не ловит джейлбрейки», хотя просто
не учился им. Стратификация = датчик мерит обобщение, не перекос разбиения.

**Метрика = детектор collapse:** recall ПАДАЕТ при росте базы → датасет выродился
(дубли, старьё, эхо). Ранний сигнал схлопывания (урок Anima).

## 🔧 HARDENING v1.1 — свод 6 аудитов (критичное, в первый код)

Шесть внешних разборов. Ниже — где сошлось у большинства (= реально) и что беру.

### КРИТИЧНО (сошлось у 4-6 аудитов)

**H1. Отдельный retrieval-эмбеддер ≠ Guard** [все 6]
Qwen3Guard — NLI-модель (entailment), её эмбеддинги слабы для поиска похожести.
Яд "ignore all previous" и benign "follow instructions carefully" → близкие у NLI,
разный вердикт. Few-shot ловил бы «похожее по стилю», не «по опасности».
→ Отдельная retrieval-модель для training_index (BGE-M3 / GTE — заточены под
cosine). Guard (NLI) и few-shot (cosine) — ОРТОГОНАЛЬНЫЕ сигналы, сливаются в вердикт.
→ Версия/хеш эмбеддера ЗАШИТА в выжимку. Pre-flight на Studio: версия не та →
fail-safe (работа без few-shot, только Guard), красный лог. Иначе смена модели =
выжимка в кашу (косинус случайный).

**H2. FPR-метрика, не только recall** [все 6]
Recall без false-positive rate → паранойя: порог занизил, recall 100%, но Guard
душит норму. → Held-out CLEAN set (тоже 70/30) + измерять FPR при том же пороге.
Целевая F-beta (β=2, recall важнее) ИЛИ F1. Порог подбирается авто на held-out,
не фиксируется навечно. Отчёт показывает ОБЕ цифры (recall + FPR).

**H3. Dedup + приближённый индекс** [5 из 6]
Сотни/тысячи эмбеддингов → линейный косинус медленный, near-duplicates раздувают.
→ FAISS HNSW/IVF или центроиды кластеров (не плоский список). При импорте: если
cos≥0.99 к существующему — не дублировать (счётчик частоты семейства, не новый эталон).
Лимит exemplars на тип атаки.

**H4. Откат/версионирование выжимки + revoke** [4 из 6]
Добавил вектор в боёвку → нет пути отозвать. Ошибся меткой / атака устарела →
выжимка шумит. → Каждый вектор несёт source_hash (16 симв, не раскрывает яд).
Отзыв: пометить в labels как revoked → ПЕРЕСБОРКА выжимки (rebuild, не append) →
боевая база обновляется. Версии выжимки (последние N), откат к согласованной с разметкой.

**H5. Канонизация хеша шире NFKC** [4 из 6]
NFKC мало. Один яд с лишним пробелом/регистром/zero-width → другой хеш → дубль-призрак.
→ Канонизатор: NFKC + удалить control-chars (category 'C') + zero-width + strip +
collapse пробелов в один + (опц.) lowercase. Хеш от «сжатой» строки.

**H6. Лимиты/TTL на _incoming/** [4 из 6]
Garak за ночь нагенерит 50k → диск кончится → автоген упадёт молча.
→ TTL: >14 дней без решения → в archive/ (не удалять). Лимит файлов/размера →
пауза автогена + Telegram-алерт. Лёгкий пред-фильтр: бинарь/картинку/дубль отсеять.

### СИЛЬНОЕ (2-3 аудита, беру)

**H7. suspect_clean — статус, не бинарь** [4 из 6]
Guard сработал на clean ≠ сразу «ошибка» (бывает FP на легитимном). → Статусы:
clean / poison / **suspect_clean** / needs_review. Сработал Guard на clean → НЕ
в clean и НЕ в poison, а в `_suspicious_clean/` на ручной пересмотр. Меньше ложных
отбрасываний.

**H8. Audit-log смены меток** [4 из 6]
Таблица label_conflicts (или поля prev_label, changed_date, supersedes, reason).
История «было clean, стало poison» — детектор дрейфа ТВОЕГО понимания. Раз в месяц
агрегат: повторные конфликты = плохая нормализация / твоё определение яда плывёт.

**H9. Сборку выжимки выполнять на Air, не на Studio** [аудит 5, тонко и верно]
Если скрипт сборки читает poison на Mac Studio с воткнутого SSD — это исполнение
с потенциально грязного носителя на боевой машине. → Сборка на Air (грязная зона),
на Studio едет ГОТОВЫЙ статичный .npy (числа), прошедший проверку формата/размера.
Границу пересекает массив, не процесс чтения ядов.

**H10. Latency budget + confidence-gate** [аудит 6]
- few-shot не должен класть API: поиск >порога latency → деградировать (откл. few-shot,
  оставить Guard). Top-K (3-5 ближайших), не вся база в контекст.
- Alert fatigue: дёргать тебя на gate ТОЛЬКО при высокой уверенности Guard.
  Низкая уверенность → не блок, а флаг needs_review_later в очередь, не прерывая поток.

### В ЗАМЕТКУ (known limitation, не блокер)

- **Эмбеддинг-инверсия:** эмбеддинг детерминирован от текста; при утечке выжимки +
  модели теоретически частично восстановим (Morris 2023). Практическая изоляция,
  НЕ криптографическая. Снижает риск на порядки, гарантии нет.
- **macOS-метаданные на SSD:** read-only монтирование при чтении + `.metadata_never_index`.
  Мелочь в скрипт, не архитектура.
- **Manifest версий** (эмбеддер/Guard/схема/counts/build-time) — чтоб понять, отчего
  recall сдвинулся (данные? модель? порог?).
- **Temporal split** рядом со стратификацией: train на старых, test на новых —
  честнее ловит новые формы, не «зубрёжку архива».

### ОТВЕРГНУТО (оверкилл / чужой масштаб)

Аппаратный USB-выключатель, external threat-intel парсинг, TextAttack adversarial,
канареечный «10% трафика» (нет трафика, 1 юзер), homoglyphs как зависимость,
dual-verifier вторым safety-классификатором (ShieldGemma) — в дальний бэклог,
не первый код.

## ❓ ЧЕГО НЕ ДЕЛАЕМ (границы, чтоб не расползлось)

- НЕ дообучаем веса Guard (это вариант б — в бэклоге, когда few-shot упрётся)
- НЕ держим живой канал Air↔Studio (только съёмный диск, физический зазор)
- НЕ кладём яды в боевую память (только эмбеддинги-выжимку)
- НЕ доверяем меткам внутри файлов (только БД по хешу)
- НЕ пускаем автоген напрямую в датасет (только через _incoming/ + твой gate)

## Что напишу в коде — ПО КУСКАМ (каждый прогоняется до следующего)

**Кусок 1 — ядро полигона** `training_sandbox.py`:
- канонизатор (NFKC+control+zero-width+collapse) → стабильный sha256 [H5]
- training_labels.db: статусы clean/poison/suspect_clean/needs_review [H7]
- импорт-валидация: хеш+метка+конфликт→стоп-вопрос, label_conflicts audit [H8]
- структура папок: _incoming/ (TTL+лимит [H6]), clean/, poison/, _suspicious_clean/ [H7]

**Кусок 2 — индекс + метрика** (на грязной стороне):
- изолированный retrieval-индекс (отдельный эмбеддер, мок в контейнере) [H1, H3]
- dedup при импорте (cos≥0.99) [H3]
- evaluator: стратиф.+temporal split 70/30, held-out recall + FPR, F-beta, авто-порог [H2]

**Кусок 3 — сборщик выжимки** (на Air → .npy на Studio):
- poison→эмбеддинги→.npy с manifest (версия эмбеддера/хеш) [H1, H9]
- revoke→rebuild, версии выжимки [H4]
- dry-run режим (показать что уедет, не писать)

**Кусок 4 — боевая интеграция** в security.py:
- FewShotMatcher: грузит .npy, pre-flight версии эмбеддера→fail-safe [H1]
- Top-K + latency budget→деградация [H10]
- добавочный сигнал к Guard (max(nli, few_shot)), не замена; cos_sim в лог
- прогон: яд НЕ течёт в боёвку; ловит похожую атаку; suspect_clean ловится;
  конфликт→стоп; recall+FPR считаются; версия-mismatch→fail-safe

Всё под мок-эмбеддер в контейнере, реальные модели — на твоём железе.
