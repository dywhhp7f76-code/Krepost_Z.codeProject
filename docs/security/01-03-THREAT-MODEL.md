# Модель угроз

Threat Model A и другие модели угроз. Данный документ описывает основные векторы атак, слои защиты и модель доверия системы КРЕПОСТЬ.

## Threat Model A: Закрытая система

Основная модель угроз исходит из того, что система не смотрит наружу — у неё нет исходящих сетевых запросов к внешним сервисам. Все атаки приходят через единственную точку входа: **пользовательский ввод**.

Это означает, что атакующий ограничен текстовым каналом и не может эксплуатировать сетевые уязвимости, SSRF или supply chain-атаки.

## 4 слоя защиты

Система использует каскадную фильтрацию, где каждый последующий слой ловит то, что пропустил предыдущий:

1. **Layer 1 — Regex Pre-filter:** быстрая детекция известных паттернов атак (base64-блоки, системные промпты, подозрительные escape-последовательности). Работает за микросекунды, отсекает очевидные атаки.
2. **Layer 2 — Guard Model:** нейросетевой классификатор (Qwen3Guard-Gen-4B), обученный на adversarial-примерах. Оценивает семантику запроса и присваивает score безопасности.
3. **Layer 3 — FewShot Classifier:** классификатор на основе few-shot примеров из Hopfield Memory. Использует эмбеддинг-сходство с известными атаками для выявления новых вариаций.
4. **Layer 4 — Output Post-Processor:** фильтрация выходных данных модели. Детекция утечек PII, проверка на наличие вредоносного контента в ответе, KVEraser для удаления harmful spans.

## Векторы атак

Система защищает от следующих типов атак:

- **Prompt Injection** — внедрение инструкций в пользовательский ввод для изменения поведения модели
- **Base64 Encoding** — обфускация вредоносного контента через кодирование
- **Homoglyph Bypass** — замена символов визуально похожими (кириллица/латиница, Unicode-подстановки)
- **Chat-template Smuggling** — эксплуатация формата чат-шаблонов для инъекции системных промптов
- **XML/CDATA Injection** — использование XML-конструкций для обхода парсинга
- **Zero-width Characters** — вставка невидимых Unicode-символов для обхода regex-фильтров

## DoS-защита

- **Per-session rate limiting** — ограничение количества запросов на сессию для предотвращения ресурсного исчерпания
- Запросы, превышающие лимит, отклоняются с соответствующим кодом ошибки

## Защита от утечек

- **PII Masking** — автоматическое маскирование персональных данных в ответах модели (email, телефоны, адреса, номера документов)
- **Leakage Detection** — детекция попыток извлечения внутренних данных системы (системные промпты, конфигурация, пути к файлам)

## Модель доверия: 3 зоны

Все данные в системе классифицируются по уровню доверия:

1. **Trusted Vault** — проверенные, верифицированные данные из собственной базы знаний. Полное доверие при RAG-поиске.
2. **Ingested** — данные, прошедшие карантин и базовую проверку, но не получившие полного аудита. Используются с ограничениями.
3. **Training** — adversarial-данные, яды, тренировочные примеры. Никогда не используются в production-ответах. Физически изолированы на съёмном SSD.
