> `KREPOST_ENABLE_MEMORY_ROUTER`. Бой на Studio — после re-ingest / миграции
> `metadata.domain`. CrossEncoder reranker — опционально.

> Канон Phase 4: [_handoff/HIERARCHICAL_DOMAIN_RAG_SPEC.md](HIERARCHICAL_DOMAIN_RAG_SPEC.md)

> Phase 3 router+retrieve **не** закрывает Phase 4.

## Цель

Память Крепости не одним раздувающимся индексом, а несколькими доменными
хранилищами. Роутер выбирает нужные, retrieval достаёт, reranker отсеивает,
основной LLM отвечает на просеянном. Основная модель НЕ ищет и НЕ фильтрует —
получает готовый контекст.

## Поток (строгий порядок)

```
запрос → роутер (какие домены) → retrieval (достать куски из выбранных)
       → reranker (отсеять лучшее) → сборка контекста → основной LLM (ответ)
```

## Компоненты и требования

### 1. Доменные хранилища ("хранители")
- Несколько отдельных индексов по доменам (старт: папки на диске, домен = папка).
- Каждый умеет только: "релевантно мне / нет" + вернуть пути к кускам.
- НЕ содержат модели, НЕ генерируют, НЕ суммируют. Только индекс + адреса.
- Границы доменов задаются вручную на старте (жёсткий список). Авто-создание
  новых доменов — отдельная фича на потом, сейчас НЕ делать.

### 2. Роутер
- Вход: запрос пользователя. Выход: список доменов для опроса (может быть >1).
- Механизм старт: классификация по эмбеддингу (e5-base, уже в проекте) +
  явные метки/папки где есть. Смешанный запрос → несколько доменов, это норма.
- Лёгкий, без вызова основного LLM.

### 3. Retrieval ("курьер")
- Достаёт куски из доменов, выбранных роутером. Параллельно. Тупой перенос.

### 4. Reranker ("вахтёр") — ОБЯЗАТЕЛЬНЫЙ клапан нагрузки
- Маленькая reranker-модель (bge-reranker-v2-m3 или аналог).
- Вход: запрос + все найденные куски. Выход: top-N самых релевантных.
- Смысл: в контекст основного LLM идёт ТОЛЬКО отобранное, не всё найденное.
  Это то, что держит нагрузку под контролем при нескольких сработавших доменах.

### 5. Основной LLM ("голова")
- Получает собранный отобранный контекст. Формулирует ответ. Всё.
- Один голос, одна модель — доменные хранилища НЕ отвечают сами (иначе стилевой разнобой).

## Управление (харнесс)

- Оркестрацию ведёт харнесс: он вызывает шаги в порядке выше, основной LLM
  будит последним, на готовом контексте.
- Управление памятью железа: reranker + embedder + основная модель на 64GB
  Mac Studio живут впритык. Харнесс должен уметь держать лёгкое загруженным,
  а тяжёлое поднимать по требованию. Заложить в дизайн сразу.

## Нагрузочный профиль (для проверки, что реализовано верно)

- хранители/роутер/retrieval: near-zero
- reranker: доли секунды, лёгкий
- основной LLM: ~90%+ веса, всегда один вызов на просеянном

Если в реализации основная модель вызывается несколько раз на домен или
получает всё найденное без reranking — архитектура собрана НЕВЕРНО.

## Что НЕ делать в первой версии

- Авто-создание доменов
- Доменных агентов-моделей (хранители остаются тупыми индексами)
- Врезку в существующий Orchestrator без отдельного PR и верификации
